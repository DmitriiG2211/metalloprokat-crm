# Промпт для Codex: сервис ИИ-анализа телефонных звонков менеджеров

## Роль

Ты — ведущий full-stack разработчик, системный архитектор, DevOps-инженер и специалист по внедрению ИИ в бизнес-процессы.

Нужно самостоятельно спроектировать и создать рабочий закрытый веб-сервис для коммерческой организации, занимающейся продажей металлопроката. Менеджеры ежедневно совершают холодные звонки потенциальным клиентам, которые чаще всего ещё не работали с компанией.

В компании примерно 5 менеджеров. Каждый совершает более 100 звонков в день. Общий объём — около 500 звонков ежедневно, иногда 1 000 и более. Сервис должен принимать большие пакеты записей, расшифровывать их, анализировать через ИИ и показывать руководителю объективные отчёты по каждому менеджеру и отделу в целом.

Не создавай только демонстрационный интерфейс или набор моков. Нужен рабочий MVP с реальной серверной логикой и возможностью дальнейшего масштабирования.

---

# 1. Цели сервиса

Руководитель должен иметь возможность:

1. Загружать сотни и тысячи аудиозаписей одним пакетом.
2. Сопоставлять звонки с менеджером, датой, клиентом и номером телефона.
3. Получать расшифровку каждого звонка с таймкодами.
4. По возможности разделять реплики менеджера и клиента.
5. Оценивать звонок по единой настраиваемой системе критериев.
6. Видеть рейтинг менеджеров не только по количеству звонков, но и по качеству и результатам.
7. Находить шаблонные фразы, слабые места, ошибки и упущенные возможности.
8. Понимать, почему холодные звонки не приводят к заявкам.
9. Получать рекомендации по улучшению скрипта и работы каждого менеджера.
10. Сравнивать периоды, менеджеров, результаты и версии скриптов.
11. Экспортировать отчёты в XLSX, CSV и PDF.

Компания продаёт трубы, листы, круги, балки, швеллеры, уголки, арматуру, нержавеющий и цветной металл, металлоизделия, резку и изготовление по чертежам.

Типичная цель холодного звонка: выйти на закупщика или ЛПР, выявить потребность, получить контакт, разрешение отправить прайс/КП и договориться о следующем действии. Не оценивай звонок только по факту продажи.

# 2. Массовая загрузка

Пользователь создаёт партию, указывает название, период, отдел и комментарий, затем загружает:

- отдельные файлы;
- множество файлов;
- ZIP-архив;
- аудио вместе с CSV/XLSX метаданными.

Поддержать MP3, WAV, M4A, OGG, OPUS, AAC, FLAC и по возможности AMR.

Одна партия должна поддерживать минимум 1 000 файлов. Обработка выполняется в фоне. После перезагрузки страницы прогресс не пропадает.

Для партии показывать:

- всего файлов;
- ожидают;
- обрабатываются;
- расшифрованы;
- проанализированы;
- завершены с предупреждением;
- завершены с ошибкой;
- процент выполнения.

Статусы pipeline:

1. uploaded;
2. validating;
3. preprocessing;
4. transcribing;
5. diarizing;
6. awaiting_analysis;
7. analyzing;
8. completed;
9. completed_with_warning;
10. failed;
11. cancelled.

# 3. Метаданные и менеджеры

Поддержать варианты сопоставления:

1. Менеджер указан в имени файла.
2. Менеджер указан в XLSX/CSV.
3. Один менеджер назначается всей партии.
4. Ручная корректировка отдельных звонков.
5. В будущем — API телефонии.

Пример таблицы:

| filename | manager | client_phone | client_company | call_date | duration | external_call_id |
|---|---|---|---|---|---|---|

Сделай настраиваемые правила разбора имени файла, не привязывайся к одному шаблону.

Пример:

`2026-06-13_10-35-22_Иванов_79991234567.mp3`

---

# 4. Расшифровка аудио

Расшифровка должна быть отдельным модулем, независимым от LLM-анализа.

Экономный базовый вариант:

- FFmpeg для нормализации;
- `faster-whisper` для локальной расшифровки;
- Silero VAD или аналог для удаления длинных пауз;
- русский язык по умолчанию;
- CPU и NVIDIA GPU;
- настраиваемая Whisper-модель;
- сохранение таймкодов и сегментов;
- хранение технической информации о качестве распознавания.

Создай интерфейс `TranscriptionProvider` и реализации:

- `local_faster_whisper`;
- `mock` только для тестов;
- интерфейс-заглушку для внешнего Speech-to-Text API.

Не отправляй аудио напрямую в Timeweb AI Agent. Используй Timeweb для смыслового анализа уже готового текста.

Опциональная диаризация:

- Speaker 1 / Speaker 2;
- попытка определить менеджера и клиента;
- ручная смена ролей;
- честное отображение confidence;
- поддержка `pyannote.audio` или другого провайдера;
- диаризация выключена по умолчанию, чтобы не увеличивать расходы;
- если разговор записан по отдельным стереоканалам, использовать каналы как более точный источник ролей.

# 5. Интеграция с ИИ

Основной провайдер — Timeweb Cloud AI Agent или AI Gateway через API.

Поддержать:

- OpenAI-совместимый API;
- при необходимости нативный API Timeweb;
- Bearer Token;
- API URL, agent ID, model и token только в переменных окружения;
- timeout;
- retry с exponential backoff;
- rate limit;
- ограничение параллельных запросов;
- журнал ошибок;
- повторный анализ;
- смену провайдера без переписывания бизнес-логики.

Создай интерфейс `LLMProvider` и реализации:

- `timeweb_agent`;
- `openai_compatible`;
- `mock` для тестов.

Запрещено:

- хранить API-ключ во frontend;
- вшивать секреты в Git;
- подменять реальный анализ случайными значениями;
- считать задачу успешной при ошибке API.

# 6. Экономия токенов и денег

Объём звонков большой, поэтому реализуй двухуровневый анализ.

## Уровень 1 — каждый звонок

В LLM отправляется очищенная расшифровка, метаданные, длительность, критерии и контекст бизнеса. Ответ — только строгий JSON.

## Уровень 2 — сводный анализ

Для дневного, недельного и месячного отчёта не отправляй повторно все расшифровки. Используй структурированные результаты отдельных звонков, короткие цитаты, статистику и репрезентативную выборку.

Обязательно:

- кеш по хешу текста и версии промпта;
- повторный анализ только при изменении текста, критериев или версии промпта;
- разбиение длинных звонков на части;
- лимит длины контекста;
- учёт входных и выходных токенов;
- приблизительная стоимость;
- дневной и месячный лимит;
- предупреждение о приближении к лимиту;
- возможность остановить автоматический анализ;
- ограничение параллельности;
- не анализировать автоответчики, пустые и технические звонки полной дорогой моделью.

# 7. Критерии оценки звонка

Критерии и их веса редактируются в административной панели. Минимальный набор:

1. Приветствие.
2. Представление менеджера и компании.
3. Понятная причина звонка.
4. Уверенная и естественная речь.
5. Отсутствие грубости и давления.
6. Отсутствие чрезмерной шаблонности.
7. Умение удержать внимание в первые 15–30 секунд.
8. Выяснение, закупает ли клиент металлопрокат.
9. Выявление потребности.
10. Уточнение продукции и сроков закупки.
11. Выход на закупщика/ЛПР.
12. Получение имени, телефона или email.
13. Работа с возражениями.
14. Открытые вопросы.
15. Умение слушать и не перебивать, если это можно определить.
16. Персонализация.
17. Конкретная польза для клиента.
18. Предложение отправить прайс/КП.
19. Фиксация следующего шага.
20. Назначение повторного звонка.
21. Корректное завершение.
22. Соблюдение скрипта.
23. Удачные и неудачные отступления от скрипта.
24. Ложные обещания и рискованные формулировки.
25. Общая вероятность продолжения контакта.

Каждый критерий содержит:

- score;
- weight;
- applicable;
- comment;
- evidence quote;
- timestamp;
- confidence.

Не штрафовать за этап, который объективно не мог состояться: моментальный сброс, неверный номер, автоответчик, закрытая компания и т.п.

Для каждого звонка сформировать:

- итоговый балл 0–100;

- краткое резюме;
- сильные стороны;
- ошибки;
- упущенные возможности;
- главную причину результата;
- этап потери клиента;
- настроение и заинтересованность клиента;
- результат звонка;
- следующий шаг;
- наличие договорённости;
- возражения;
- рекомендации менеджеру;
- рекомендации руководителю;
- хорошие фразы;
- шаблонные/слабые фразы;
- улучшенные варианты формулировок;
- флаг ручной проверки;
- confidence.

Категории результата:

- клиент заинтересован;
- отправить прайс;
- отправить КП;
- получен контакт закупщика;
- назначен повторный звонок;
- есть текущая заявка;

- другое.

Разделять технический дозвон, содержательный разговор, целевой контакт, контакт с ЛПР и положительный следующий шаг.

# 9. Строгий JSON и защита от галлюцинаций

Создай Pydantic-модель и JSON Schema. Минимальный формат:

```json
{
  "summary": "string",
  "call_quality_score": 0,
  "quality_category": "good",
  "confidence": 0.0,
  "call_outcome": {
    "code": "send_price",
    "label": "Отправить прайс",
    "confidence": 0.0
  },
  "meaningful_conversation": true,
  "decision_maker_reached": false,
  "need_identified": false,
  "next_step_agreed": true,
  "next_step": {
    "type": "send_email",
    "description": "Отправить прайс",
    "due_date": null
  },
  "client_sentiment": "neutral",
  "client_interest": "medium",
  "objections": [],
  "criteria": [],
  "strengths": [],
  "mistakes": [],
  "missed_opportunities": [],
  "template_phrases": [],
  "manager_recommendations": [],
  "supervisor_recommendations": [],
  "manual_review_required": false,
  "manual_review_reasons": []
}
```

Требования:

- валидировать ответ через Pydantic;
- ограниченный repair/retry при невалидном JSON;
- хранить raw response отдельно;
- не сохранять невалидный ответ как итоговый;
- запрещать модели выдумывать факты;
- важные выводы подтверждать цитатой и таймкодом;
- при недостатке данных возвращать `insufficient_data`.

Базовый системный промпт:

> Ты — строгий и объективный руководитель отдела холодных продаж металлопроката и эксперт по речевой аналитике. Анализируй только предоставленную расшифровку и метаданные. Не выдумывай реплики, результаты, договорённости, контакты или потребности. Различай качество работы менеджера и фактический итог звонка. Учитывай, что звонок холодный. Не снижай оценку за этап, который объективно не мог состояться. Каждый важный вывод подтверждай короткой цитатой и таймкодом, если он доступен. Возвращай только JSON по указанной схеме.

Системный промпт должен быть версионируемым и редактируемым администратором.

# 10. Причины низких продаж

Сервис должен отвечать руководителю:

- почему большое число звонков не приводит к заявкам;
- кто звонит формально;
- какие вступления чаще связаны с отказом;
- на каком этапе теряется клиент;
- какие возражения менеджеры не умеют обрабатывать;
- как часто выходят на закупщика;
- сколько разговоров заканчивается без следующего шага;
- кто делает много звонков, но мало полезных контактов;
- кто делает меньше звонков, но имеет лучшие результаты;
- какие товары вызывают интерес;
- какие дни и часы эффективнее;
- какие формулировки используют лучшие менеджеры;
- как изменить скрипт;
- чему обучить каждого менеджера.

Не делать категоричных выводов по маленькой выборке. Всегда показывать размер выборки и confidence. Корреляцию не называть доказанной причиной.

# 12. Шаблонные фразы и речевая аналитика

Найти:

- частые вступления;
- фразы, после которых клиент чаще завершает разговор;
- фразы, связанные с положительным продолжением;
- слова-паразиты;
- длинные монологи;
- чрезмерно сложные предложения;
- неопределённые и негативные формулировки;
- давление и оправдания;
- вопросы, провоцирующие простой ответ «нет»;
- отсутствие конкретики и следующего шага.

Раздел «Библиотека фраз»:

- фраза;
- контекст;
- положительный/негативный эффект;
- частота;
- связанные результаты;
- менеджеры;
- рекомендуемая замена.

# 13. Скрипты и ручная проверка

Администратор может:

- создавать версии скрипта;
- добавлять этапы;
- задавать обязательные и желательные фразы;
- запрещать формулировки;
- задавать критерии и веса;
- загружать инструкции и примеры;

Руководитель может:

- изменить результат;
- исправить роли говорящих;
- скорректировать оценку;
- отметить ошибку ИИ;
- оставить комментарий;
- отметить хороший или плохой пример;
- подтвердить/отклонить рекомендацию.

Не заявлять автоматическое дообучение модели, если оно реально не реализовано.

# 14. Пользователи и роли

Роли:

- Администратор — все настройки, пользователи, интеграции, лимиты, аудит.

RBAC реализовать на backend, а не только скрытием элементов интерфейса.
